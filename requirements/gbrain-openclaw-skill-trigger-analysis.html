⚡ Architecture Analysis

GBrain + OpenClaw
Skill Trigger Resolution

How OpenClaw discovers gbrain skills, injects them into the agent context, and how the two-layer resolver picks the right skill on every message.

1

The Installation Bridge — openclaw.plugin.json

gbrain declares itself to OpenClaw as a bundle-plugin via openclaw.plugin.json. This file is the handshake between the two systems and wires two things simultaneously:

{
  "name": "gbrain",
  "family": "bundle-plugin",
  "mcpServers": {
    "gbrain": { "command": "./bin/gbrain", "args": ["serve"] }
  },
  "skills": [
    "skills/capture", "skills/query", "skills/signal-detector",
    "skills/brain-ops", /* ... 43 total ... */
  ],
  "shared_deps": [
    "skills/conventions",
    "skills/_AGENT_README.md",   // ← the routing contract
    "skills/_brain-filing-rules.md",
    "skills/_output-rules.md"
  ],
  "resolver": "RESOLVER.md"
}

⚡ MCP Server

./bin/gbrain serve registers as an MCP server, exposing the brain tools the LLM calls during skill execution: get_page, put_page, add_link, add_timeline_entry, etc.

📚 Skills List

The 43 skill directories gbrain contributes to the workspace. Each one will be scanned by OpenClaw's runtime loader and injected into the agent's system prompt.

On every AlphaClaw startup, syncWorkspaceSkills (called from syncBootstrapPromptFiles) re-syncs lib/setup/skills/ → workspace/skills/, guaranteeing skills survive container restarts. Shared dependencies (_AGENT_README.md, RESOLVER.md, conventions/) are copied alongside.

2

Skill Discovery — OpenClaw's Runtime Scanner

On session start, OpenClaw calls loadWorkspaceSkillEntries → loadSkillsFromDirSafe, which walks <workspaceDir>/skills/ and runs loadSingleSkillDirectory for each subdirectory:

// Simplified from OpenClaw dist/workspace-DkDBQCx-.js
function loadSingleSkillDirectory(params) {
  // 1. Read SKILL.md (symlink-rejected, size-capped)
  const raw = readSkillFileSync(path.join(params.skillDir, "SKILL.md"));
  if (!raw) return null;

  // 2. Parse YAML frontmatter
  const frontmatter = parseFrontmatter(raw);

  // 3. Extract name + description (ONLY these two go into the prompt)
  const name = frontmatter.name?.trim() || path.basename(params.skillDir);
  const description = frontmatter.description?.trim();
  if (!name || !description) return null;

  // 4. Resolve invocation policy (disable-model-invocation flag)
  const invocation = resolveSkillInvocationPolicy(frontmatter);

  return {
    name, description,
    filePath: path.resolve(skillFilePath),
    baseDir:  path.resolve(params.skillDir),
    source:   params.source,
    disableModelInvocation: invocation.disableModelInvocation
  };
}
⚠
Only name and description reach the prompt. The triggers:, tools:, mutating:, and writes_pages: frontmatter fields are extracted for policy enforcement internally but are not injected into the LLM context directly. The full frontmatter remains available for shouldIncludeSkill filtering.
3

Prompt Injection — The <available_skills> Block

buildWorkspaceSkillsPrompt calls formatSkillsForPrompt, which appends an XML block to the agent's system prompt on every new session:

The following skills provide specialized instructions for specific tasks.
Use the read tool to load a skill's file when the task matches its description.
When a skill file references a relative path, resolve it against the skill directory
(parent of SKILL.md) and use that absolute path in tool commands.

<available_skills>
  <skill>
    <name>capture</name>
    <description>Save any thought or content into the brain via one CLI
      command. The single human-facing entrypoint that replaces "put_page vs
      commit-then-sync vs autopilot-wait" with one command that just works.</description>
    <location>/workspace/skills/capture/SKILL.md</location>
  </skill>
  <skill>
    <name>signal-detector</name>
    <description>Always-on ambient signal capture. Fires on every message
      to detect original thinking and entity mentions.</description>
    <location>/workspace/skills/signal-detector/SKILL.md</location>
  </skill>
    <!-- ... all 43 skills ... -->
</available_skills>

💡 Key insight

This XML block is the primary dispatch surface. The LLM sees every skill's name + description and uses semantic reasoning to decide which <location> to read. The instruction "use the read tool to load a skill's file" means the full SKILL.md body — including Phases, Contract, and Anti-Patterns — is only consumed after the skill is selected, not upfront.

4

The Two-Layer Trigger System

Trigger resolution uses two complementary layers. Neither layer alone is sufficient — together they handle both typed commands and natural language intent.

Layer 1 — OpenClaw Runtime

Description-based, implicit matching

  • LLM scans <available_skills> descriptions
  • Semantic / fuzzy reasoning over natural language
  • Coarse — catches intent even without exact phrase match
  • Injected instruction: "use read tool when task matches description"

Layer 2 — gbrain Agent Instruction

Frontmatter triggers, explicit phrase matching

  • LLM reads _AGENT_README.md (scaffolded shared dep)
  • Instructed to walk triggers: arrays in SKILL.md frontmatter
  • Substring match as baseline; semantic similarity on top
  • Precise — catches exact user phrases from MECE trigger lists

Each skill's frontmatter declares its explicit trigger vocabulary:

---
name: capture
triggers:
  - "capture this"
  - "save this thought"
  - "remember this"
  - "drop this in the inbox"
  - "save to brain"
writes_pages:
  - "inbox/*"
---
ℹ
RESOLVER.md is documentation, not dispatch. Before v0.36, RESOLVER.md was a managed routing table the agent parsed directly. From v0.36+, _AGENT_README.md explicitly tells the agent: "Do NOT look for a managed-block table inside RESOLVER.md or AGENTS.md. Routing lives in frontmatter now." RESOLVER.md is auto-maintained as a human-readable reference.

Worked example — "save this to brain":

Layer 1 (description scan): capture → "Save any thought or content..." → candidate ✓ ingest → "Route content to specialized..."→ candidate ✓ brain-ops → "Brain-first lookup, read-enrich" → no match Layer 2 (frontmatter triggers): capture/SKILL.md triggers: ["save this thought", "save to brain"] → "save to brain" substring hit → strong match ✓ ingest/SKILL.md triggers: ["ingest this", "save this to brain"] → "save this to brain" substring hit → weaker match Disambiguation: capture is more specific to single-thought saves → invoke capture ★
5

Disambiguation Rules

When multiple skills match, RESOLVER.md defines the tiebreak order. These rules are embedded in the agent's instruction set via the scaffolded file:

Priority Rule Example
1 — First Specificity wins — prefer the most specific skill meeting-ingestion beats ingest
2 URL content type routing — if a URL is present, classify it YouTube URL → media-ingest; article link → idea-ingest
3 Entity type routing — person/company name present Check if enrich or query is the better fit
4 Chaining is explicit — follow Phases section of matched skill Each skill names what to chain into next
5 — Last When in doubt → invoke the choice-gate pattern ask-user/SKILL.md — present 2–4 options, stop until user replies
6

Always-On Skills — Pre-Dispatch Parallel Execution

Two skills bypass trigger matching entirely. They fire on every inbound message, spawned in parallel, non-blocking — they never delay the user-facing response:

signal-detector

Ambient entity/signal capture. Detects original thinking, entity mentions, and notable concepts in every message. The brain compounds via this skill.

brain-ops

Brain-first lookup on any brain read/write/lookup/citation. Enforces the "check brain before external API" invariant across all skills.

✓
These are declared in RESOLVER.md under "Always-on (every message)" and confirmed by signal-detector's description: "Fires on every message." They are invariants, not options — routing them through trigger matching would defeat their purpose as ambient background processes.
7

Skill Execution — Read-Then-Follow Cycle

Once a skill is selected, execution is a strict read-then-follow cycle:

1. LLM calls: read("/workspace/skills/capture/SKILL.md")
   ← uses OpenClaw's built-in read tool

2. Reads ## Contract — confirms guarantees match the task

3. Executes ## Phases step by step:
   Phase 1: detect input type
   Phase 2: call MCP tool → gbrain capture "..."
   Phase 3: verify receipt

4. Calls gbrain MCP tools as needed:
   get_page, put_page, add_link, add_timeline_entry, ...

5. Consults _brain-filing-rules.md on every write
   ← "file by PRIMARY SUBJECT, not format"

6. Follows chaining instructions if Phases says to:
   "for meeting data, chain into skills/meeting-ingestion/SKILL.md"
MCP ToolPurpose
get_pageRead a page from the brain DB
put_pageWrite/update a brain page (rewrite State, never append)
add_timeline_entryAppend to entity timeline (reverse-chronological)
add_linkCreate typed link between entities (Iron Law back-links)
get_backlinksRetrieve back-links for a page
searchHybrid RAG search (embedding + keyword + rerank)
put_raw_dataStore raw API responses for provenance
sync_brainTrigger brain sync to disk

9

Key Architectural Decisions

Why frontmatter triggers, not RESOLVER.md?

RESOLVER.md (pre-v0.36) required manual synchronization between the routing table and each skill's actual intent — they could drift. With frontmatter triggers (v0.36+), each skill owns its routing contract. RESOLVER.md is generated documentation; frontmatter is the source of truth. MECE validation at skill creation time enforces no trigger overlap across the entire skill set.

Why Layer 1 (description) AND Layer 2 (triggers)?

Layer 1 is coarse but zero-cost — the LLM already scans <available_skills>. Layer 2 is precise — explicit trigger strings reduce ambiguous multi-skill matches. Together they handle typed commands ("capture this") and natural language intent ("I'd like to save this article I'm reading") without requiring an exact phrase.

Why always-on skills bypass trigger matching?

Signal detection and brain-first lookup are invariants, not options. Routing them through trigger matching means they only fire when the user explicitly invokes them — defeating their purpose as ambient background processes that make the brain compound over time. They are spawned in parallel and never block the user-facing response.

The disable-model-invocation escape hatch

Skills that are purely programmatic (e.g., publish, which generates static HTML with zero LLM calls) set disable-model-invocation: true in frontmatter. OpenClaw's resolveSkillInvocationPolicy picks this up and routes them to slash-command or direct CLI invocation rather than LLM description matching.
