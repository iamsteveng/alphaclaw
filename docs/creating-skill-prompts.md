# Creating a Skill from a Prompt

A guide for turning a self-contained LLM prompt into a reusable OpenClaw skill.

## What makes a good skill prompt?

A skill prompt is a **self-contained system prompt** that:

1. **Transforms the agent's behavior** for a specific use case
2. **Has a clear trigger** (when should it activate?)
3. **Defines an interaction protocol** (how does the agent talk to the user?)
4. **Is reusable** across sessions and contexts

The Pentimento analysis skill is the reference example used throughout this guide.

## Step-by-step

### 1. Write the prompt first

Start with a plaintext prompt. Test it by pasting it into an LLM. Iterate until the behavior is right. Don't worry about frontmatter or file structure yet — nail the behavior first.

A good skill prompt has:

- **A name and identity** — what the agent IS when running this skill
- **An internal framework** — the mental model the agent uses to think
- **A protocol** — the exact interaction pattern with the user
- **Quality rules** — guardrails that prevent bad behavior

### 2. Add YAML frontmatter

Every skill file needs frontmatter. Minimum required fields:

```yaml
---
name: your-skill-name          # hyphenated, lowercase
description: One sentence on what it does. Used in skill listings.
triggers:                      # Array of phrases that activate this skill
  - "trigger phrase one"
  - "trigger phrase two"
---
```

Optional fields:

```yaml
metadata:                      # OpenClaw-specific metadata
  openclaw:
    emoji: "🔍"                # Emoji shown in skill listings
    requires:                  # External dependencies
      bins: ["some-cli-tool"]
mutating: true                 # Does this skill write to the brain?
writes_pages: true             # Does it create/modify brain pages?
writes_to:                     # Directories it writes to
  - research/
version: 0.1.0
```

### 3. Structure the body

The body is a standard markdown document. A proven structure:

```
# Skill Name

One-paragraph summary of what this does and when to use it.

## The Framework
The internal model/mental framework. This is what the agent uses to think.
Define layers, stages, or phases clearly. Each should have:
- A name
- A question it answers
- A method or technique it uses

## Interaction Protocol
The exact sequence of interactions. Be specific:
- What does the agent do first?
- What does the user see?
- How does the user respond?
- What happens next?
Use numbered phases if the flow has branches.

## Quality Rules
Guardrails. What must the agent NEVER do? What makes a good vs bad
execution? Include anti-patterns.

## Philosophical Lineage (optional)
If your framework synthesizes established traditions, cite them.
This adds credibility and gives users entry points for deeper study.
```

### 4. Define triggers

Triggers are how OpenClaw knows to activate your skill. They're substring-matched against user messages.

Rules for good triggers:
- **Be specific enough** to avoid false positives (don't use "analyze" alone)
- **Be general enough** to catch natural variations
- **Include common synonyms** for your skill's action
- **Test in conversation** — would a normal message accidentally trigger this?

Example from Pentimento:
```yaml
triggers:
  - pentimento                    # Direct name call
  - analyze in layers             # Natural language
  - layered analysis              # Synonym
  - drill down                    # Common phrase during use
  - unpack this                   # Natural variation
  - what are the principles behind  # Question format
```

### 5. File placement

```
skills/<skill-name>/SKILL.md
```

The directory name must match the `name` field in frontmatter. The file must be `SKILL.md` (case-sensitive).

Skills live under your workspace: `~/.openclaw/workspace/skills/`

### 6. Test with a real session

Run the skill against real content. Things to check:
- Does it activate on the right triggers?
- Does it stay silent on unrelated messages?
- Does the interaction protocol feel natural?
- Are the quality rules preventing bad outputs?
- Does it handle edge cases (thin content, ambiguous content, very long content)?

### 7. Register with the skillpack (auto)

If you're using gbrain's skillpack system, the skill is auto-discovered. The manifest in `AGENTS.md` is generated by `skillpack-check` — **do not hand-edit it**.

Run `skillpack-check` to validate and register new skills.

## Anatomy of a SKILL.md (reference)

Using Pentimento as the example:

```markdown
---
name: pentimento
description: Layered content analysis using the Pentimento framework...
triggers:
  - pentimento
  - analyze in layers
  - layered analysis
  - drill down
  - unpack this
  - what are the principles behind
---

# Pentimento Analysis

[One-paragraph identity statement]

## The Five Layers

### Layer 4: Axioms — "What must be true?"
[Description + method]

### Layer 3: Principles — "What rules govern this?"
[Description + method]

### Layer 2: Structures — "What systems are at play?"
[Description + method]

### Layer 1: Phenomena — "What is actually happening?"
[Description + method]

### Layer 0: Implications — "What should one DO?"
[Description + quality criteria for actions]

## Interaction Protocol

### Phase 1: Receive Content
### Phase 2: Deliver Actions
### Phase 3: Drill Down
### Phase 4: Continue

## Quality Rules

[Guardrails and anti-patterns]

## Philosophical Lineage

[Synthesis of traditions cited]
```

## Common pitfalls

- **Too many triggers** → skill activates constantly. Keep it to 5-8.
- **Protocol too rigid** → feels robotic. Leave room for natural variation.
- **No quality rules** → agent drifts. Always include guardrails.
- **Framework too abstract** → agent can't apply it. Each layer/phase needs a concrete method.
- **Actions too generic** → "read more" or "think about it" are not actions. Each action must be a single, specific, doable thing.
- **Missing the close** → after sending actions, always tell the user what to do next.
- **Frontmatter typos** → `triggers:` not `trigger:`, `name:` must match directory name.

## When to make a skill vs a one-off prompt

| Make a skill when... | Keep as one-off when... |
|---|---|
| You'll use it more than once | It's for a single session |
| It has an interaction protocol | It's a static instruction set |
| Others might benefit from it | It's highly personal/temporary |
| It transforms agent behavior significantly | It's a minor style tweak |
| You want to trigger it by name | You'll just paste it in |
